Storage Architecture

The application uses in-memory storage for simplicity and demonstration purposes. All expense data is stored in JavaScript arrays within the server's memory, making it perfect for development and testing environments.

Data Schema

The expense data follows a well-defined schema with the following structure:

Field Type Required Description
_id String Yes Unique identifier (auto-generated)
title String Yes Expense title/name
amount Number Yes Expense amount (positive number)
category String Yes Expense category (predefined options)
date String Yes Expense date (ISO format)
description String No Optional expense description
createdAt String Yes Record creation timestamp

üîó API Architecture

RESTful Endpoints

The backend provides a complete set of RESTful API endpoints for expense management:

GET /api/expenses

Retrieve all expenses with optional filtering by category, start date, and end date

GET /api/expenses/:id

Retrieve a specific expense by its unique identifier

POST /api/expenses

Create a new expense record with validation

PUT /api/expenses/:id

Update an existing expense record

DELETE /api/expenses/:id

Delete an expense record permanently

GET /api/expenses/stats/summary

Retrieve statistical summary including total amount, count, and category breakdown

Request/Response Format

All API endpoints use JSON format for data exchange. The server validates incoming data and returns appropriate HTTP status codes with descriptive error messages.

üîÑ Data Flow & Integration

User Interaction Flow

  1. User Action: User interacts with the interface (add, edit, delete, search)
  2. Frontend Processing: JavaScript validates input and prepares API request
  3. API Call: HTTP request sent to Express server endpoint
  4. Server Processing: Express routes handle request, validate data, and update storage
  5. Response: Server returns JSON response with updated data
  6. UI Update: Frontend processes response and updates interface

State Management

The application maintains state through:

  • Global Variables: Frontend stores current expenses, stats, and UI state
  • Server Memory: Backend maintains expense data in JavaScript arrays
  • Real-time Sync: Automatic data refresh after CRUD operations

Error Handling

Comprehensive error handling includes:

  • Client-side validation before API calls
  • Server-side validation with descriptive error messages
  • Network error handling with user-friendly notifications
  • Graceful degradation when server is unavailable

üöÄ Deployment & Setup

Local Development

  1. Install Node.js (version 14 or higher)
  2. Extract project files and navigate to directory
  3. Run npm install to install dependencies
  4. Run npm start to start the server
  5. Access application at http://localhost:5000

Production Considerations

For production deployment, consider:

  • Replace in-memory storage with persistent database (MongoDB, PostgreSQL)
  • Add user authentication and authorization
  • Implement data backup and recovery mechanisms
  • Add comprehensive logging and monitoring
  • Configure HTTPS and security headers
